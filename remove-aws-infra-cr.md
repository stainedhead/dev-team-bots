# Change Request: Remove AWS Infrastructure — Local Single-Binary Runtime

**CR ID:** remove-aws-infra-cr  
**Date:** 2026-05-06  
**Status:** Draft — Analysis  

---

## 1. Problem Statement

The current `boabot` runtime is tightly coupled to six AWS services (SQS, SNS, S3, S3 Vectors, DynamoDB, Secrets Manager) and assumes deployment on ECS/Fargate with an ECR image. This creates several barriers:

- **High operational friction.** A developer cannot run the system without an active AWS account, correct IAM roles, provisioned queues, tables, and buckets. The CDK stack must be deployed before a single bot can start.
- **Cost at idle.** AWS charges for every SQS poll, DynamoDB read, and S3 request even when the team is doing nothing.
- **Deployment complexity.** Each bot is a separately containerised process coordinated via external message queues. Adding a new bot requires a CDK stack update, ECR push, and ECS task definition change.
- **No local-first story.** There is no way to run the full multi-bot team on a laptop for development or evaluation without mocking every AWS adapter.

The desired end state is a single self-contained binary that runs the full bot team locally, with AWS providers remaining available as optional plugins for teams that want them.

---

## 2. Desired State

### 2.1 Single Executable

`boabot` becomes the single entry point for the entire team. On startup it:

1. Reads `team.yaml` to discover enabled bot configurations.
2. Starts the orchestrator bot as the primary goroutine.
3. Spawns each additional enabled bot as an in-process goroutine, each with its own config, `SOUL.md`, tool set, and memory namespace.
4. Manages bot lifecycle (start, health-check, restart on panic) without any external process manager.

No separate container images. No ECS. One binary, one `go run` or compiled executable.

### 2.2 Local Memory

Memory files are stored on the local filesystem, not S3. The default root is `<app-dir>/memory/`, configurable via `memory.path` in config or `BOABOT_MEMORY_PATH` env var. Each bot gets its own subdirectory: `memory/<bot-name>/`.

### 2.3 In-Process Messaging

SQS and SNS are replaced by Go channels and an in-process event bus. Bots communicate by sending typed `Message` values through buffered channels rather than over the network. The `MessageQueue` and `Broadcaster` domain interfaces remain unchanged — only the infrastructure adapter changes.

### 2.4 In-Process Vector Search

The S3 Vectors semantic search adapter is replaced by an embedded vector store: embeddings are stored as `[]float32` slices on disk (alongside memory files), and cosine similarity search runs in-process at query time. Embeddings are generated by whichever LLM provider is configured (using its embedding endpoint), with a BM25 keyword-search fallback when no embedding endpoint is available.

### 2.5 Budget Tracking — In-Process

DynamoDB budget counters are replaced by in-memory counters (per bot, per run) with optional persistence to a local JSON file (`memory/<bot-name>/budget.json`). Counters survive restarts via this file.

### 2.6 Secrets and Configuration

Secrets Manager is replaced by:
- Config file fields (for non-sensitive values).
- Environment variables (for API keys and sensitive credentials, following `BOABOT_<KEY>` naming).
- Optional: a local credentials file at `~/.boabot/credentials` (mode 0600) for persistent key storage, following the same pattern as `boabotctl`.

### 2.7 LLM Providers — Expanded

Three first-class providers, all optional and configurable:

| Provider | Type key | Transport |
|---|---|---|
| AWS Bedrock | `bedrock` | AWS SDK (existing) |
| OpenAI-compatible | `openai` | HTTP to configured endpoint (existing) |
| Anthropic Claude SDK | `anthropic` | `github.com/anthropics/anthropic-sdk-go` (new) |

Provider selection remains per-bot via `models.providers` in `config.yaml`. A bot that sets `type: anthropic` uses the Claude API directly (no Bedrock hop).

## 3. Component-by-Component Analysis

### 3.1 Removed — AWS Infrastructure Packages

| Package | Current role | Replacement |
|---|---|---|
| `infrastructure/aws/sqs` | A2A message queue | `infrastructure/local/queue` — buffered `chan Message` per bot |
| `infrastructure/aws/sns` | Team-wide broadcast | `infrastructure/local/bus` — in-process pub/sub event bus |
| `infrastructure/aws/s3` | Memory file storage | `infrastructure/local/fs` — `os.ReadFile`/`os.WriteFile` under configured path |
| `infrastructure/aws/s3vectors` | Semantic vector search | `infrastructure/local/vector` — cosine similarity over in-memory/on-disk float32 vectors |
| `infrastructure/aws/dynamodb` | Budget counter persistence | `infrastructure/local/budget` — in-memory counters + JSON file flush |
| `infrastructure/aws/secretsmanager` | Credential storage | Environment variables + optional `~/.boabot/credentials` file |
| `infrastructure/aws/secrets` | Secret loader at startup | Removed; replaced by env-var loader |
| `infrastructure/aws/bedrock` | LLM invocation | Remains as optional provider; not required if `anthropic` or `openai` is configured |
| `boabot-team/cdk/` | CDK stacks for all infrastructure | Removed entirely |

### 3.2 New — Local Infrastructure Packages

**`infrastructure/local/queue`**
- Implements `domain.MessageQueue` using a `chan ReceivedMessage` per bot.
- Receipt handles are UUIDs assigned on enqueue; `Delete` is a no-op (channel delivery is at-most-once).
- `Send` routes to the correct bot's channel by queue URL (replaced by bot name in local mode).

**`infrastructure/local/bus`**
- Implements `domain.Broadcaster`.
- Maintains a registry of subscriber channels. `Broadcast` fans out to all registered bots.
- Thread-safe via `sync.RWMutex`.

**`infrastructure/local/fs`**
- Implements `domain.MemoryStore`.
- `Write`/`Read`/`Delete` map directly to `os.WriteFile`/`os.ReadFile`/`os.Remove` under `<memory.path>/<bot-name>/`.
- Creates directories on first write.

**`infrastructure/local/vector`**
- Implements `domain.VectorStore`.
- Embeddings stored as binary files (`<memory.path>/<bot-name>/vectors/<key>.vec`).
- `Search` loads all vectors into memory, computes cosine similarity, returns top-k results.
- Acceptable for team sizes up to ~100k stored vectors; beyond that, a pluggable HNSW index can be swapped in later.

**`infrastructure/local/budget`**
- Implements `domain.BudgetTracker`.
- Counters held in a struct with `sync/atomic` fields.
- Flush writes to `<memory.path>/<bot-name>/budget.json`; startup loads from this file.
- No external dependency.

**`infrastructure/anthropic`**
- Implements `domain.ModelProvider` using `github.com/anthropics/anthropic-sdk-go`.
- Supports model selection via `model_id` in provider config (e.g. `claude-sonnet-4-6`).
- Maps `InvokeRequest` → Anthropic Messages API → `InvokeResponse`.
- Streams by default; exposes `StopReason` and `TokenUsage` from the API response.
- API key read from `ANTHROPIC_API_KEY` env var or `anthropic_api_key` in the credentials file.

### 3.3 Modified — Bot Lifecycle (New Application Layer)

A new `application/team` package handles multi-bot orchestration within the single process:

- **`TeamManager`** — reads `team.yaml`, instantiates each enabled bot's config, wires up its adapters, and calls `Agent.Start()` in a goroutine.
- **`BotRegistry`** — maintains a map of running bot instances; used by the local bus to route messages.
- **Panic recovery** — each bot goroutine has a `recover()` wrapper; a crashed bot is restarted with exponential back-off (matches existing worker-thread behaviour).
- **Graceful shutdown** — `TeamManager.Shutdown()` sends `ShutdownMessage` to all bots and waits for their goroutines to exit.

`cmd/boabot/main.go` changes from wiring a single bot to instantiating `TeamManager` and blocking until signal.

### 3.4 Modified — Configuration Schema

`config.yaml` schema changes:

```yaml
# REMOVED — entire aws section
# aws:
#   region: ...
#   sqs_queue_url: ...
#   sns_topic_arn: ...
#   private_bucket: ...
#   team_bucket: ...
#   dynamodb_budget_table: ...

# ADDED — memory section
memory:
  path: ./memory          # default: <binary-dir>/memory; supports ~ expansion
  vector_index: cosine    # cosine (default) | hnsw (future)

# MODIFIED — models section gains anthropic provider type
models:
  default: claude
  providers:
    - name: claude
      type: anthropic     # new
      model_id: claude-sonnet-4-6
    - name: bedrock-claude
      type: bedrock       # existing, now optional
      model_id: anthropic.claude-3-5-sonnet-20241022-v2:0
      region: us-east-1
    - name: openai
      type: openai        # existing
      model_id: gpt-4o
      endpoint: https://api.openai.com/v1

# UNCHANGED — bot, orchestrator, tools, budget, context sections
```

Top-level team config (`team.yaml`) is unchanged structurally; per-bot `config.yaml` files drop the `aws` section.

### 3.5 New — Scheduled GitHub Memory Backup

Memory files replace S3 as the persistence layer, but S3 implicitly provided durability and off-site backup. This section adds an equivalent guarantee using a user-owned GitHub repository.

**Design: treat the memory directory as a git repository.**

On first run (or on `boabotctl memory init`), the `memory.path` directory is initialised as a git repository and a GitHub remote is added. The backup scheduler periodically stages all changes, commits with a timestamped message, and pushes to the configured remote. Restore is `git clone <repo> <memory.path>`.

**Domain interface — `MemoryBackup`:**
```go
type MemoryBackup interface {
    // Backup commits all pending changes and pushes to the remote.
    Backup(ctx context.Context) error
    // Restore clones or pulls the remote into the configured memory path.
    Restore(ctx context.Context) error
    // Status returns the number of uncommitted changes since the last backup.
    Status(ctx context.Context) (BackupStatus, error)
}

type BackupStatus struct {
    LastBackupAt   time.Time
    PendingChanges int
    RemoteURL      string
}
```

**Infrastructure adapter — `infrastructure/github/backup`:**
- Uses `go-git` (`github.com/go-git/go-git/v5`) — pure Go git implementation, no `git` binary required.
- Authentication via a GitHub personal access token (PAT) or a GitHub App token, read from `BOABOT_BACKUP_TOKEN` env var or the credentials file.
- Commit author set from config (`backup.github.author_name`, `backup.github.author_email`).
- On `Backup()`: `git add -A`, `git commit -m "backup: <RFC3339 timestamp>"`, `git push`. If nothing has changed since the last commit, the push is skipped.
- On conflict (remote has diverged): pull with rebase before pushing. Local changes always win on conflict.

**Application layer — `application/backup` (scheduled use case):**
- A `ScheduledBackupUseCase` wraps `MemoryBackup` and is driven by the existing `scheduler` package.
- Default schedule: every 30 minutes (configurable via cron expression).
- Backs up the entire `memory.path` tree — memory files, vector embeddings, budget state.
- Emits a log line and an internal metric on each backup (success/failure, files changed, duration).
- Failure is non-fatal: a failed backup logs an error and retries on the next tick.

**Configuration additions:**
```yaml
backup:
  enabled: false                        # opt-in
  schedule: "*/30 * * * *"             # cron; default every 30 min
  github:
    repo: "https://github.com/owner/boabot-memory.git"
    branch: main                        # default: main
    author_name: "BaoBot"
    author_email: "boabot@example.com"
    # token read from BOABOT_BACKUP_TOKEN env var
```

**`boabotctl` additions:**
- `boabotctl memory backup` — trigger an immediate backup.
- `boabotctl memory restore` — clone/pull from the configured remote into `memory.path`.
- `boabotctl memory status` — show last backup time and pending change count.

**What is backed up:**
- All files under `memory.path/` including: memory key/value files, vector index files (`.vec`), budget state (`budget.json`).
- Files matching `.gitignore` patterns in `memory.path/.gitignore` are excluded (default: none).

**Restore path on first startup:**
If `backup.enabled: true` and `memory.path` is empty or absent, `boabot` automatically attempts a `Restore()` before starting the team. This means deploying to a new machine requires only a config file and a backup token.

### 3.6 Embedder — LLM-Backed with BM25 Fallback

The `domain.Embedder` interface already exists. In local mode:

1. If the configured provider exposes an embedding endpoint (OpenAI `/v1/embeddings`, Anthropic text embedding when available), the `Embedder` adapter calls it.
2. If no embedding endpoint is available (Bedrock invocation-only), a `BM25Embedder` falls back to sparse BM25 keyword vectors. Vector similarity still works but is lexical rather than semantic.
3. The vector store is provider-agnostic — it stores whatever `[]float32` the embedder produces.

## 4. What Does NOT Change

- All domain interfaces (`MessageQueue`, `Broadcaster`, `MemoryStore`, `VectorStore`, `Embedder`, `BudgetTracker`, `ModelProvider`, etc.) are unchanged. The adapter swap is purely at the infrastructure layer.
- The agent runtime loop (`run_agent.go`, `execute_task.go`, `context_manager.go`, etc.) is unchanged.
- The orchestrator HTTP server and Kanban board are unchanged — they can still be exposed if `orchestrator.enabled: true`.
- `boabotctl` is unchanged — it communicates with the orchestrator REST API regardless of transport backend.
- TDD and Clean Architecture boundaries are unchanged.
- Bedrock remains a supported provider for teams already on AWS.

## 6. Open Questions

1. **Embedding strategy:** Should the system require at least one embedding-capable provider at startup, or is BM25 fallback always acceptable? If semantic search is a core feature, the CR should make embedding a hard requirement.

2. **Vector index scale:** Cosine brute-force is fine for small memory stores (<100k vectors). At what scale does a team actually operate? If memory stores grow large, an HNSW index (e.g. `hnswgo`) should be planned from the start rather than retrofitted.

3. **Bot isolation:** Goroutines share heap. A memory-leaking or runaway bot can starve others. Should each bot have a memory cap enforced at the runtime level (e.g. via a token-budget goroutine), or is the existing `BudgetTracker` sufficient?

4. **Credentials file format:** Should `~/.boabot/credentials` follow the AWS credentials file format (ini sections per profile) or a simpler YAML/env format? Consistency with `boabotctl`'s credential storage matters here.

5. **Bedrock removal completeness:** Is `infrastructure/aws/bedrock` to be deleted, or retained as an optional provider that simply isn't the default? The CR currently retains it. If the intent is zero AWS imports in the default binary, Bedrock must be moved behind a build tag.

6. **Plugin architecture for providers:** "Plugin support for Claude SDK" could mean (a) a built-in adapter compiled into the binary, or (b) a Go plugin (`plugin.Open`) loaded at runtime. Option (a) is simpler and testable; option (b) enables third-party providers without recompilation. Which is intended?

7. **`boabot-team/cdk/` fate:** The CDK stacks deploy shared infra (VPC, ECS cluster) and per-bot task definitions. Removing them is clean for local mode, but teams using AWS for hosting (even without the messaging layer) may want a stripped-down CDK stack. Should a minimal "compute-only" CDK stack be retained?

8. **Backup repo visibility:** Should the GitHub backup repo be private by default, or is that the user's responsibility? Should `boabotctl memory init` offer to create and configure the repo automatically via the GitHub API?

9. **Backup scope granularity:** The current design backs up the full `memory.path` tree in one commit per tick. For large teams with high write rates, this could produce noisy git history. Should per-bot subdirectory repos be an option, or is a single team repo always sufficient?

10. **Auto-restore on startup:** Automatically pulling from the backup remote on first startup is convenient but risky if the remote has stale or corrupted data. Should this be opt-in (`backup.restore_on_empty: true`) rather than the default behaviour?

## 7. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| In-process bots share heap — one bad bot can OOM the process | Medium | BudgetTracker + memory-limit config; goroutine isolation via recover() |
| BM25 fallback produces lower-quality memory retrieval than semantic search | Medium | Document the trade-off; make embedding provider configuration clearly required for semantic search |
| Removing DynamoDB persistence means budget counters are lost on crash (before flush) | Low | Short flush interval (default 10s) + write-ahead log pattern if needed |
| Anthropic SDK API surface may change faster than Bedrock | Low | Wrap behind `ModelProvider` interface; adapter is isolated |
| Large teams (10+ bots) may hit goroutine scheduler pressure on small hardware | Low | Benchmark before GA; goroutines are cheap, but model invocations are the real bottleneck |
| GitHub PAT leaked via config or logs | Medium | Token read exclusively from env var or credentials file (never config.yaml); never logged |
| `go-git` push fails silently during network outage | Low | Retry with exponential back-off; emit metric; alert via orchestrator notification channel |
| Backup repo grows unboundedly — git history accumulates every 30-min commit forever | Low | Document `git gc` + shallow-clone restore; optionally squash history older than N days |
